FASE 1 — Activación de MFA
Rol: GENERAL_ADMIN · SUCURSAL_ADMIN
1.1
Ir a “Configuración” en el panel admin (entrada del sidebar con icono engrane). Solo visible para roles GENERAL_ADMIN y SUCURSAL_ADMIN.
1.2
Click “Activar MFA” → se abre Dialog de 3 pasos (estado step local en SettingsView).
1.3
Paso 1: escanear QR con app autenticadora (Google Authenticator, Authy, 1Password) o copiar secret base32 manualmente. Origen: POST /api/auth/mfa/setup → devuelve otpauth_uri + qrDataUrl.
1.4
Paso 2: ingresar código TOTP de 6 dígitos en InputOTP → click “Verificar y activar”. Backend: POST /api/auth/mfa/verify valida con authenticator.verify() y marca mfaEnabled = true, mfaEnrolledAt = now().
1.5
Paso 3: guardar los 10 backup codes en lugar seguro (botón “Descargar códigos” genera .txt vía Blob + URL.createObjectURL + <a download>).
1.6
Click “He guardado los códigos, finalizar” → MFA queda activo. Badge verde mfaEnabled = true persistido en DB. Cierre del dialog y refresh de estado MFA.

FASE 2 — Login con MFA
Rol: Usuario con MFA activo
2.1
Ir a la URL del sistema: https://control-asistencia-v22.vercel.app
2.2
Ingresar email + password → click “Iniciar sesión”. Frontend envía POST /api/auth/login con { email, password }.
2.3
El sistema detecta mfaEnabled = true en validateCredentials() → responde 200 { needsMfa: true, email }. Frontend cambia a vista “Verificación en dos pasos” (icono Shield).
2.4
Abrir app autenticadora → copiar código TOTP actual (6 dígitos, válida 30 s · params digits=6, period=30, algorithm=SHA1).
2.5
Ingresar código en InputOTP → click “Verificar y entrar”. Frontend re-envía POST /api/auth/login con { email, password, mfaToken }.
2.6
Sesión creada con mfaVerified: true en JWT NextAuth → acceso al panel. Cookie next-auth.session-token (httpOnly, SameSite=strict, secure en prod, maxAge 8 h, rotación 30 min).

FASE 3 — Recuperación con backup code
Rol: Usuario con MFA activo, sin dispositivo TOTP
3.1
Ir a la URL del sistema.
3.2
Ingresar email + password → click “Iniciar sesión”.
3.3
El sistema pide MFA (paso 2 de login) → click “Usar código de respaldo en su lugar” (toggle useBackupCode = true; el InputOTP se reemplaza por input de texto).
3.4
Ingresar uno de los 10 backup codes guardados (formato XXXX-XXXX-XXXX-XXXX, case-insensitive, normalizado a UPPERCASE antes de comparar).
3.5
Click “Verificar y entrar” → backend recorre JSON.parse(mfaBackupCodesHash) comparando con bcrypt.compare(). Si matchea: backup code consumido (one-time use — el hash correspondiente se remueve del array y se persiste mfaBackupCodesHash actualizado).
3.6
Sesión creada con mfaVerified: true → acceder al panel. Acción recomendada: re-configurar MFA con nuevo dispositivo (Fase 4 → Fase 1) para reponer el código consumido.

FASE 5 — Seguridad técnica
Técnico Referencia para admins técnicos
5.1 · Secret
Secret TOTP encriptado con AES-256-GCM en DB. Clave derivada de NEXTAUTH_SECRET vía SHA-256. Formato almacenado: IV(12) ‖ AuthTag(16) ‖ ciphertext en base64.
5.2 · Backup
Backup codes hasheados con bcrypt (cost 10). Almacenados como JSON array en mfaBackupCodesHash. Solo se muestran una vez durante el setup.
5.3 · JWT
NextAuth JWT firmado (JWE) con NEXTAUTH_SECRET. Claim mfaVerified: boolean incluido en token y sesión. maxAge = 8 h, rotación cada 30 min.
5.4 · Lockout
Tras 5 intentos fallidos → cuenta bloqueada 15 min (lockedUntil = now + 15 min). failedLoginAttempts se resetea en login exitoso.
5.5 · One-time
Backup codes son one-time use: al consumirse, el hash se remueve del array en DB. Si se agotan, se recomienda re-enrolar MFA para regenerarlos.
